## 一、架构决策记录 (ADR)

### ADR-001：统一响应包装 `Result<T>`

- **决策**：废弃 `Ret<T>`，全系统统一使用 `Result<T>`（`{code, message, data}`）
- **原因**：两套包装并存导致前端需要判断两种格式，增加维护成本
- **影响范围**：8 个文件（Controller、Service、GlobalExceptionHandler、JwtInterceptor）

### ADR-002：统一分页格式 `PageResult<T>`

- **决策**：所有分页接口统一返回 `{records, total, page, size, pages}`，不再使用 `Map<String, Object>`
- **原因**：契约不一致，管理端和读者端分页格式不同
- **影响范围**：AdminArticleController、ArticleController、ArticleReadService

### ADR-003：Controller 只返回 VO，不返回 Domain 对象

- **决策**：所有 Controller 返回 VO/DTO，不在接口层暴露领域对象
- **原因**：领域对象包含业务行为，序列化可能导致意外副作用（如值对象 toString 格式）
- **影响范围**：TagController、CategoryController（已完成）；CommentController、LinkController（待修复）

### ADR-004：Application 层禁止直接注入 Mapper

- **决策**：ApplicationService 只通过 Repository 接口访问数据，不直接注入 `*Mapper`
- **原因**：违反 DDD 依赖方向 `application → domain ← infrastructure`
- **验证**：TagApplicationService、CategoryApplicationService 已修复为通过 Repository

## 二、DDD 落地经验

### ✅ 做对了的事

1. **Repository 接口定义在 domain 层**：3 个 Context 的 Repository 接口位置正确，扩展新方法时遵循了模式（如 `countDraftsByAuthor()`）
2. **值对象不可变**：`@final class` + `private final` 字段设计正确
3. **构造器注入**：全系统统一使用构造器注入，利于单元测试
4. **Bounded Context 拆分**：article/user/comment 拆分合理，互不依赖

### ⚠️ 容易犯的错误

1. **Application 层直接调 Mapper**：开发者在"统计文章数"这种看似简单的需求面前，选择了最直接的方式（注入 Mapper），而没有遵循 Repository 模式。**教训**：所有数据访问必须通过 Repository，即使只是一行 `selectCount()`
2. **值对象序列化**：领域对象（含值对象）直接返回给前端时，Jackson 可能调用 `toString()` 产生 `@{id=1; name=Java}` 格式。**教训**：Controller 层必须转换为 VO，值对象手动拆解为基本类型
3. **前后端契约大小写不一致**：客户端发送 `"APPROVE"`，服务端只认 `"approve"`。**教训**：契约文档必须与实际代码对齐，或后端做大小写不敏感处理

## 三、值对象序列化陷阱

本次发现的核心陷阱：**领域对象中的值对象被 Jackson 序列化为 `toString()` 格式**。

| 现象 | 根因 | 表现 |
|------|------|------|
| `category: "@{id=1; name=前端; ...}"` | CategoryVO 未正确组装，直接传了 domain 对象 | 前端无法解析为对象 |
| `author: "@{id=1; nickname=博主; ...}"` | AuthorVO 未正确组装 | 同上 |
| `tags: ""` | 标签列表未正确转换为数组 | 前端渲染空白 |

**根治方案**：
- Controller 层严格返回 VO
- ApplicationService 的 `toVO()` 方法手动拆解值对象（如 `slug.value()` → String）
- 前端加防御性检查：`Array.isArray()` / `typeof === 'string'`

## 四、前端防御性编程经验

| 场景 | 防御措施 | 示例 |
|------|---------|------|
| API 响应解包 | 不假设嵌套层级，检查数组格式 | `Array.isArray(result) ? result : []` |
| 值对象序列化 | 判断字段是字符串还是对象 | `typeof row.slug === 'string' ? row.slug : row.slug?.value` |
| 枚举值大小写 | 使用小写统一 | `'approve' \| 'reject'` 而非 `'APPROVE' \| 'REJECT'` |
| 可选字段 | 使用可选链 | `author?.avatar \|\| ''` |

## 六、对后续任务的建议

1. **新功能开发前**：先修复 P0/P1 遗留问题（预估 1-2 小时）
2. **新增 Controller**：严格返回 VO，模板化 toVO() 转换方法
3. **新增 Mapper 查询**：通过 Repository 接口暴露，不直接在 ApplicationService 注入
4. **前后端契约**：任何接口变更先更新 `docs/api-contract-v2.md`，双方据此独立开发
5. **代码审查**：每个 PR 必须通过四层架构合规性检查
