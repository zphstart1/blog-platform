## 第一阶段：JVM & 线上排查（Week 1-4）

**这是面试最高频考点，也是最大的短板，优先补齐。**

### Week 1：JVM 内存模型

- 搞清楚堆的分区：Eden / Survivor / Old / Metaspace
- GC 触发时机、Minor GC vs Full GC 的区别
- 推荐资料：《深入理解Java虚拟机》第2、3章（不用全读）

### Week 2：GC 日志和工具实操

在本地 Docker 起一个 Spring Boot 应用，加上以下 JVM 参数：

```bash
-Xms256m -Xmx256m
-XX:+PrintGCDetails -XX:+PrintGCDateStamps
-Xloggc:/tmp/gc.log
```

- 用 `jstat -gcutil <pid> 1000` 观察 GC 频率
- 用 `jmap -heap <pid>` 看内存占用
- 用 GCEasy 网站上传分析 gc.log

### Week 3：CPU 飙高排查流程

掌握这个固定套路，背下来，能手写：

```bash
1. top                         # 找到 Java 进程 PID
2. top -Hp <pid>               # 找到占用CPU最高的线程 TID
3. printf "%x\n" <TID>         # 转成16进制
4. jstack <pid> | grep <16进制TID> -A 30  # 看线程在干什么
```

自己写一个死循环或死锁的 demo，然后用这个流程排查一遍。

### Week 4：OOM 排查

- 自己写内存泄漏 demo（比如静态 List 一直 add）
- 触发 OOM 后导出堆快照：

```bash
jmap -dump:format=b,file=heap.hprof <pid>
```

- 用 MAT（Eclipse Memory Analyzer）分析，找到泄漏对象

**阶段产出**：能在面试中讲清楚"某次线上 CPU 飙高，我是怎么排查的"（哪怕是自己模拟的经历）

## 第二阶段：技术栈深度挖掘（Week 5-11）

### MySQL（Week 5-6）

| 主题 | 核心问题 |
|------|---------|
| 索引 | B+树结构、为什么不用B树、联合索引最左匹配 |
| 执行计划 | EXPLAIN 各字段含义，怎么判断索引失效 |
| 事务 | MVCC 原理、RR 隔离级别怎么解决幻读 |
| 锁 | 行锁/表锁/间隙锁，死锁如何产生和检测 |
| 慢SQL | 你工作中遇到过慢SQL吗？怎么优化的？ |

**实操**：开启慢查询日志，用 `EXPLAIN` 分析一条业务里真实的 SQL。

### Redis（Week 7）

| 主题 | 核心问题 |
|------|---------|
| 数据结构 | ZSet 为什么用跳表不用红黑树 |
| 持久化 | RDB vs AOF 区别，项目里用哪个，为什么 |
| 缓存问题 | 穿透/雪崩/击穿的区别和解决方案 |
| 分布式锁 | Redisson 原理，为什么不能直接用 SETNX |

**结合业务**：项目里哪里用了 Redis？能不能讲清楚为什么这么用？

### RocketMQ（Week 8）

| 主题 | 核心问题 |
|------|---------|
| 可靠性 | 消息丢失的三个阶段，如何保证不丢 |
| 幂等性 | 消费者怎么保证幂等的？用的什么方案 |
| 顺序消息 | 全局顺序 vs 局部顺序，实现原理 |
| 延迟消息 | XXL-Job 和 MQ 延迟消息的选型区别 |

### Spring Boot（Week 9）

- Bean 生命周期（面试经典题）
- 循环依赖如何解决（三级缓存）
- `@Transactional` 失效的 6 种场景（结合自己踩过的坑来讲）
- Spring AOP 原理：JDK动态代理 vs CGLIB

### Elasticsearch（Week 10-11）

| 主题 | 核心问题 |
|------|---------|
| 基础概念 | Index/Document 对应 MySQL 的什么？Shard 和 Replica 的作用 |
| 写入流程 | 数据写入 ES 的完整链路（translog → buffer → segment） |
| 搜索原理 | 倒排索引是什么，为什么搜索快 |
| 相关性打分 | BM25 算法基本原理，score 是怎么算出来的 |
| 查询类型 | `term` vs `match` 的区别（面试高频） |
| 性能问题 | 深度分页为什么慢？`search_after` 怎么解决 |
| 数据同步 | MySQL 和 ES 数据如何同步？binlog/MQ/双写方案对比 |

**面试必备**，整理这几个问题的答案：

```
1. 你们 ES 存的是什么数据？为什么不直接用 MySQL？
2. 数据怎么从 MySQL 同步到 ES 的？延迟多少？
3. 遇到过搜索慢的问题吗？怎么排查的？
4. 你们的分词器用的什么？（IK分词器基本要知道）
```

**和大模型的结合点**：ES 的 KNN 向量搜索（dense_vector + knn query）是 RAG 场景的主流方案，如果项目里用 ES 做向量检索，可以作为大模型项目亮点来讲，两个技术栈直接打通，面试中很有说服力。

## 第三阶段：大模型方向深化（Week 12-15）

**这是差异化竞争力，别人没有，要把它做成亮点。**

### 理论层

- RAG（检索增强生成）完整链路：文档切割 → Embedding → 向量存储 → 召回 → 重排 → 生成
- Function Calling / Tool Use 的实现原理
- Agent 的 ReAct 框架
- Prompt Engineering：Few-shot、Chain-of-Thought、系统提示词设计

### 工程层（Java 视角）

- Spring AI 框架基本使用
- 向量数据库选型：Milvus / PGVector / Elasticsearch KNN
- 流式输出（SSE）的实现
- 大模型调用的限流、降级、成本控制

### 项目亮点整理模板

用这个结构整理现有大模型项目经验：

```
背景：公司有什么业务痛点
方案：设计了什么方案（RAG？Agent？直接调用？）
难点：遇到了什么技术问题（幻觉/上下文长度/延迟/成本）
结果：效果如何，有没有量化数据
```
