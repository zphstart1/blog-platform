# 10-知识沉淀文档

> **负责人**：team-lead | **日期**：2026-06-16

## 本次经验总结

### 1. Markdown 渲染：单数据源优于双字段

**教训**：数据库同时存储 `content`(MD) 和 `content_html`(HTML) 是一种常见反模式。

**问题链**：
```
写入时渲染 → 渲染逻辑升级 → 老数据 HTML 不更新 → 渲染不一致
修复老数据 → 批量脚本 → 下次升级又要再跑 → 恶性循环
```

**正确做法**：参考 Hexo/Hugo/Ghost/VuePress 等行业标准——**数据库只存原始 MD，前端/构建时统一渲染**。单一数据源，永不出现新旧数据不一致。

### 2. 转义序列陷阱

**现象**：Markdown 渲染只成功了解析 `##` 开头的 `<h2>`，后续 `###` 和 `-` 保留在 `<p>` 内。

**根因**：数据库中 `content` 包含字面量 `\n` 转义序列（`\\n`），而非真正的换行符。Markdown 解析器需要真正的换行符来识别块级边界。

**排查方法**：
1. 用 `console.log(JSON.stringify(content))` 查看原始字符串
2. 对比 Node.js 端渲染结果与浏览器实际 DOM
3. 使用浏览器自动化（agent-browser/Playwright）抓取 DOM 快照

**修复**：渲染前 normalize——`content.replace(/\\n/g, '\n')`

### 3. CSS 暗色模式自动切换陷阱

**现象**：页面背景莫名其妙变黑，但代码中没有设置黑色背景。

**根因**：`github-markdown-css` 包的默认 CSS 包含 `@media (prefers-color-scheme: dark)` 媒体查询，当系统使用深色模式时自动切换。

**修复**：使用 `github-markdown-css/github-markdown-light.css`（不含深色模式）

### 4. 前端构建缓存清理

**现象**：代码改了但页面不生效，Vite 构建 hash 不变。

**根因**：`npm run build` 不会自动清理 `dist/` 目录，旧构建产物残留导致 hash 计算基于旧文件。

**修复**：
- 构建前先 `rmdir /s /q dist`
- 或 `package.json` build 脚本增加 `rimraf dist &&` 前置步骤

### 5. 浏览器自动化验证的价值

使用 `agent-browser` 工具进行 DOM 快照验证，比人工肉眼检查更可靠：

```bash
agent-browser open http://localhost:3000/article/xxx
agent-browser wait --load networkidle
agent-browser snapshot  # 查看完整 DOM 树
agent-browser close
```

## 技术方案对比

| 方案 | 优点 | 缺点 | 采用 |
|------|------|------|:--:|
| 后端渲染 (commonmark-java) | 首屏快 | 双字段同步问题 | ❌ |
| 前端渲染 (marked) | 单数据源、升级方便 | 首屏依赖 JS | ✅ |
| SSR/SSG (Nuxt/Next) | 首屏快 + 单数据源 | 架构改动大 | 未采用 |

## 推荐最佳实践

1. **Markdown 内容存储**：数据库只存原始 MD，前端用 `marked`/`markdown-it` 渲染
2. **转义序列**：写入时确保换行为真正 `\n` 而非 `\\n`，或前端渲染前统一 normalize
3. **CSS 主题**：明确指定浅色/深色主题文件，避免依赖 `prefers-color-scheme` 自动切换
4. **构建流程**：`dist` 目录构建前先清理，避免旧产物残留
5. **验证手段**：结合浏览器自动化做 DOM 级别的正确性验证
