# 线上问题排查实战手册

> 适用场景：直接登录 Linux 服务器进行问题排查  
> 每个场景均附可运行的 Demo，先制造问题再观测指标，形成肌肉记忆

---

## 一、必备基础命令速查

```bash
top                    # 综合监控，CPU/内存/进程
htop                   # top 的增强版（需安装）
vmstat 1               # 每秒输出系统资源概况
iostat -x 1            # 每秒输出磁盘 IO 详情
netstat -an            # 网络连接状态
ss -s                  # TCP 连接统计摘要
sar -n DEV 1           # 每秒网络流量统计
df -h                  # 磁盘空间使用
free -m                # 内存使用（MB单位）
ulimit -n              # 当前用户文件句柄上限
```

## 场景一：CPU 飙高

### 关注指标

| 命令 | 关注字段 | 含义 |
|------|---------|------|
| `top` | `%Cpu(s)` 中的 `us`（用户态）、`sy`（内核态） | us高=业务代码问题，sy高=系统调用过多 |
| `top` | `load average` 三个值 | 超过CPU核数说明任务在排队 |
| `vmstat 1` | `r` 列 | 等待CPU的进程数，持续>核数就是问题 |
| `top -Hp <pid>` | 找到占用高的线程 TID | 下一步用 jstack 定位代码 |

### 完整排查流程

```bash
# 第一步：找到问题进程
top
# 记录 Java 进程的 PID，按 q 退出

# 第二步：找到问题线程
top -Hp <pid>
# 记录 CPU 占用最高的线程 TID

# 第三步：TID 转 16进制
printf "%x\n" <TID>
# 输出：例如 3e2

# 第四步：jstack 定位代码
jstack <pid> | grep "3e2" -A 30
# 找到对应线程的调用栈
```

### 实战 Demo（死循环导致CPU飙高）

```java
// CpuHighDemo.java
// 运行：javac CpuHighDemo.java && java CpuHighDemo
public class CpuHighDemo {
    public static void main(String[] args) {
        // 启动一个死循环线程模拟CPU飙高
        new Thread(() -> {
            System.out.println("死循环线程启动，线程ID: " + Thread.currentThread().getId());
            long count = 0;
            while (true) {
                count++; // 死循环，不做任何IO，纯CPU消耗
            }
        }, "cpu-spike-thread").start();

        System.out.println("主线程PID: " + ProcessHandle.current().pid());
        System.out.println("程序已启动，请用 top 观察CPU使用率");

        // 主线程保持存活
        try { Thread.currentThread().join(); } catch (InterruptedException e) {}
    }
}
```

**观测步骤：**
1. 运行 Demo
2. 另开终端执行 `top`，观察 CPU 使用率飙到接近 100%（单核）
3. 执行 `top -Hp <pid>` 找到占用最高的线程
4. 执行 `printf "%x\n" <TID>` 转换
5. 执行 `jstack <pid> | grep <16进制> -A 20`
6. 看到 `cpu-spike-thread` 线程在执行死循环代码

**期望看到的 jstack 输出：**
```
"cpu-spike-thread" #12 prio=5 os_prio=0 tid=... nid=0x3e2 runnable
   java.lang.Thread.State: RUNNABLE
    at CpuHighDemo.lambda$main$0(CpuHighDemo.java:7)
```

## 场景二：内存泄漏 / OOM

### 关注指标

| 命令 | 关注字段 | 含义 |
|------|---------|------|
| `free -m` | `used` / `available` | available 持续减少说明内存在泄漏 |
| `top` | `%MEM`、`RES` 列 | RES 是进程实际占用的物理内存 |
| `vmstat 1` | `swpd` 列 | 不为0说明在用交换分区，性能会很差 |
| `jstat -gcutil <pid> 1000` | `O`（老年代使用率） | 持续增长且GC后不回落 = 内存泄漏 |

### 完整排查流程

```bash
# 第一步：观察内存趋势
watch -n 2 free -m
# 观察 available 是否持续下降

# 第二步：观察 JVM 堆内存
jstat -gcutil <pid> 2000
# 每2秒刷新一次，关注 O（老年代）是否持续增长

# 第三步：导出堆快照
jmap -dump:format=b,file=/tmp/heap.hprof <pid>

# 第四步：用 MAT 分析
# 下载 Eclipse MAT：https://www.eclipse.org/mat/
# 打开 heap.hprof → Leak Suspects Report → 查看占用内存最多的对象
```

**jstat 输出说明：**
```
S0     S1     E      O      M     CCS    YGC     YGCT    FGC    FGCT     GCT
0.00  23.45  67.89  45.23  94.56  91.23     12    0.234     2    0.456    0.690
```
- `E`：Eden 区使用率
- `O`：老年代使用率（关键！持续涨 = 泄漏）
- `YGC/FGC`：Young GC / Full GC 次数
- `FGCT`：Full GC 累计耗时（Full GC 频繁 = 严重问题）

### 实战 Demo（内存泄漏）

```java
// MemoryLeakDemo.java
// 启动参数：java -Xmx256m -Xms256m MemoryLeakDemo
// 加上GC日志：java -Xmx256m -Xms256m -XX:+PrintGCDetails -Xloggc:/tmp/gc.log MemoryLeakDemo
import java.util.*;

public class MemoryLeakDemo {
    // static 持有引用，GC 无法回收 = 内存泄漏
    private static final List<byte[]> leakList = new ArrayList<>();

    public static void main(String[] args) throws InterruptedException {
        System.out.println("PID: " + ProcessHandle.current().pid());
        System.out.println("模拟内存泄漏，每秒分配 1MB，请用 jstat 观察老年代...");

        while (true) {
            leakList.add(new byte[1024 * 1024]); // 每次 1MB
            System.out.println("已分配: " + leakList.size() + " MB");
            Thread.sleep(500);
        }
    }
}
```

**观测步骤：**
1. 运行 Demo
2. 另开终端执行 `jstat -gcutil <pid> 1000`
3. 观察 `O`（老年代）持续增长，GC 后也不降
4. 最终触发 `java.lang.OutOfMemoryError: Java heap space`
5. 如果加了 `-XX:+HeapDumpOnOutOfMemoryError` 参数，OOM时会自动导出hprof文件

## 场景三：磁盘 IO 瓶颈

### 关注指标

| 命令 | 关注字段 | 含义 |
|------|---------|------|
| `iostat -x 1` | `%util` | 磁盘繁忙度，接近100%说明IO打满 |
| `iostat -x 1` | `await` | 平均IO等待时间（ms），> 20ms 就要关注 |
| `iostat -x 1` | `r/s`、`w/s` | 每秒读写次数（IOPS） |
| `top` | `%wa`（iowait） | CPU 等待磁盘 IO 的时间占比，> 20% 说明磁盘是瓶颈 |
| `df -h` | `Use%` | 磁盘空间使用率，> 95% 服务会挂 |

### iostat 输出说明

```
Device   r/s    w/s    rkB/s    wkB/s   await  %util
sda      0.5   120.3    4.0    1923.2    45.2   89.5
```
- `await=45.2`：平均每次IO等45ms，很慢（正常SSD应该< 1ms）
- `%util=89.5`：磁盘已经89%满负荷，接近瓶颈

### 完整排查流程

```bash
# 第一步：确认是否 IO 问题
top
# 看 %wa (iowait)，如果 > 20% 说明 IO 是瓶颈

# 第二步：找到哪块磁盘有问题
iostat -x 1
# 看 %util 最高的设备

# 第三步：找到是哪个进程在狂写
iotop -o
# 需要 root 权限，-o 只显示有IO活动的进程

# 第四步：磁盘空间检查
df -h        # 看各挂载点空间
du -sh /*    # 找占用最大的目录
du -sh /var/log/*  # 通常日志是大头
```

### 实战 Demo（大量磁盘写入）

```java
// DiskIODemo.java
import java.io.*;
import java.nio.file.*;

public class DiskIODemo {
    public static void main(String[] args) throws Exception {
        System.out.println("PID: " + ProcessHandle.current().pid());
        System.out.println("开始模拟大量磁盘写入，请用 iostat -x 1 观察...");

        File file = new File("/tmp/io_test.dat");
        byte[] data = new byte[1024 * 1024]; // 1MB 数据块

        try (FileOutputStream fos = new FileOutputStream(file)) {
            int count = 0;
            while (count < 10000) {
                fos.write(data);   // 写入1MB
                fos.flush();       // 强制刷到磁盘（产生真实IO）
                count++;
                System.out.println("已写入: " + count + " MB");
                // 不sleep，持续写入
            }
        }
        System.out.println("写入完成，文件大小约 10GB");
    }
}
```

**观测步骤：**
1. 另开终端运行 `iostat -x 1`
2. 运行 Demo
3. 观察 `wkB/s`（写入速率）、`await`（等待时间）、`%util`（磁盘繁忙度）飙高
4. 同时用 `top` 观察 `%wa`（iowait）上升

## 场景四：网络连接异常

### 关注指标

| 命令 | 关注字段 | 含义 |
|------|---------|------|
| `ss -s` | TIME-WAIT 数量 | 大量TIME-WAIT说明短连接过多 |
| `netstat -an \| grep ESTABLISHED \| wc -l` | 数量 | 当前活跃连接数 |
| `netstat -an \| grep TIME_WAIT \| wc -l` | 数量 | 等待关闭的连接数 |
| `sar -n DEV 1` | `rxkB/s`、`txkB/s` | 每秒接收/发送流量 |
| `cat /proc/net/sockstat` | `sockets: used` | 系统总 socket 使用数 |

### TCP 连接状态速查

```
ESTABLISHED  正常连接，数据传输中
TIME_WAIT    连接关闭中（主动关闭方），等待2MSL后消失
CLOSE_WAIT   被动关闭方，说明程序没有关闭连接（连接泄漏！）
SYN_RECV     收到SYN但还没完成握手，大量此状态 = SYN Flood攻击
LISTEN       端口监听中
```

**重点**：大量 `CLOSE_WAIT` 是严重问题，说明代码里有连接没有关闭（资源泄漏）

### 完整排查流程

```bash
# 第一步：查看连接状态分布
ss -s
# 或者
netstat -an | awk '/^tcp/ {++state[$NF]} END {for(key in state) print key, state[key]}'

# 第二步：如果 CLOSE_WAIT 很多，找是哪个进程
netstat -anp | grep CLOSE_WAIT | awk '{print $7}' | sort | uniq -c | sort -rn

# 第三步：查看网络流量
sar -n DEV 1 5
# 观察网卡是否打满带宽

# 第四步：查看端口占用
ss -tlnp | grep 8080    # 查看8080端口的监听情况
```

### 实战 Demo（连接泄漏导致大量 CLOSE_WAIT）

```java
// ConnectionLeakDemo.java
// 模拟 HTTP 客户端不关闭连接，产生 CLOSE_WAIT
import java.net.*;
import java.io.*;
import java.util.*;

public class ConnectionLeakDemo {
    private static final List<Socket> leakSockets = new ArrayList<>();

    public static void main(String[] args) throws Exception {
        System.out.println("PID: " + ProcessHandle.current().pid());
        // 需要先启动一个服务端，可以用 nc -l 8888 或者任意TCP服务
        System.out.println("模拟连接泄漏（不关闭Socket），请先启动一个TCP服务在8888端口");
        System.out.println("观察命令：watch -n 1 'netstat -an | grep 8888'");

        for (int i = 0; i < 100; i++) {
            try {
                Socket socket = new Socket("127.0.0.1", 8888);
                leakSockets.add(socket); // 持有引用，不关闭 = 连接泄漏
                System.out.println("创建连接 #" + (i + 1) + "，不关闭");
                Thread.sleep(200);
            } catch (ConnectException e) {
                System.out.println("连接失败，请确保8888端口有服务在监听");
                break;
            }
        }

        System.out.println("已创建 " + leakSockets.size() + " 个泄漏连接，保持5分钟...");
        Thread.sleep(300_000);
    }
}
```

**观测步骤：**
1. 开另一个终端运行 `nc -l 8888`（模拟服务端）
2. 运行 Demo
3. 执行 `watch -n 1 'netstat -an | grep 8888 | wc -l'` 观察连接数增长
4. 停掉 nc，观察大量连接变为 `CLOSE_WAIT` 状态

## 场景五：文件句柄耗尽

### 关注指标

| 命令 | 含义 |
|------|------|
| `ulimit -n` | 当前用户的文件句柄上限（默认1024，生产应调大） |
| `lsof -p <pid> \| wc -l` | 某进程打开的文件句柄数量 |
| `cat /proc/sys/fs/file-nr` | 系统级别：已用/最大文件句柄数 |
| `lsof -p <pid> \| grep socket` | 查看进程打开了多少 socket |

### 完整排查流程

```bash
# 第一步：检查系统句柄上限
ulimit -n
cat /proc/sys/fs/file-max

# 第二步：查看某进程的句柄使用量
lsof -p <pid> | wc -l

# 第三步：找出哪类文件句柄最多
lsof -p <pid> | awk '{print $5}' | sort | uniq -c | sort -rn
# REG=文件 IPv4/IPv6=网络连接 FIFO=管道

# 临时调大限制（不重启）
ulimit -n 65535
```

**报错特征**：文件句柄耗尽时，应用日志通常出现：
```
java.io.IOException: Too many open files
```

### 实战 Demo（文件句柄泄漏）

```java
// FileHandleLeakDemo.java
// 先调小限制再测试：ulimit -n 100
import java.io.*;
import java.util.*;

public class FileHandleLeakDemo {
    private static final List<FileInputStream> leakStreams = new ArrayList<>();

    public static void main(String[] args) throws Exception {
        System.out.println("PID: " + ProcessHandle.current().pid());
        System.out.println("建议先执行：ulimit -n 100（调小限制，更容易触发）");
        System.out.println("观察命令：watch -n 1 'lsof -p " + ProcessHandle.current().pid() + " | wc -l'");

        // 创建临时文件用于测试
        File tempFile = File.createTempFile("leak_test", ".txt");
        tempFile.deleteOnExit();

        int count = 0;
        try {
            while (true) {
                FileInputStream fis = new FileInputStream(tempFile);
                leakStreams.add(fis); // 不关闭，产生句柄泄漏
                count++;
                if (count % 10 == 0) {
                    System.out.println("已打开文件句柄: " + count + " 个");
                }
                Thread.sleep(100);
            }
        } catch (IOException e) {
            System.out.println("触发错误！已打开 " + count + " 个句柄");
            System.out.println("错误信息：" + e.getMessage()); // Too many open files
        }
    }
}
```

## 场景六：综合排查 SOP（标准作业流程）

**收到告警或接到线上问题时，按此顺序排查：**

```
第一步：看整体概况（1分钟）
├── top                  # CPU、内存、Load
├── free -m              # 内存详情
└── df -h                # 磁盘空间

第二步：定位问题类型（2分钟）
├── CPU us高             → 走【场景一】CPU飙高流程
├── CPU wa高             → 走【场景三】磁盘IO流程
├── 内存 used 高         → 走【场景二】内存泄漏流程
├── Swap 有使用          → 内存不足，JVM堆配置或机器扩容
└── 磁盘 Use% > 90%      → 紧急清理日志或扩容

第三步：Java应用专项排查
├── jstat -gcutil <pid> 1000   # GC是否频繁
├── jstack <pid>               # 线程状态，找死锁/死循环
└── jmap -heap <pid>           # 堆内存详情

第四步：网络排查
├── ss -s                      # 连接状态分布
└── sar -n DEV 1               # 网络流量

第五步：日志确认
└── tail -f /var/log/应用.log  # 结合时间点看报错
```

## 常用命令组合备忘

```bash
# 实时监控某进程的CPU和内存
top -p <pid>

# 每秒打印系统关键指标
vmstat 1

# 找出最占内存的进程 Top10
ps aux --sort=-%mem | head -10

# 找出最占CPU的进程 Top10
ps aux --sort=-%cpu | head -10

# 查看某端口被哪个进程占用
lsof -i :8080
ss -tlnp | grep 8080

# 实时监控日志
tail -f /path/to/app.log | grep -i error

# 快速统计日志中的错误数
grep -c "ERROR" /path/to/app.log

# 查看系统最近的 OOM 记录
dmesg | grep -i "out of memory"

# 查看系统最近的错误日志
journalctl -p err --since "1 hour ago"
```

## 面试话术参考

**面试官：你遇到过CPU飙高的线上问题吗？怎么排查的？**

> 遇到过类似的情况，排查步骤是：首先用 top 确认是哪个 Java 进程 CPU 使用率异常，
> 然后用 `top -Hp <pid>` 找到占用最高的线程 TID，转成16进制后用 jstack 输出线程栈，
> 定位到具体的代码位置。我们那次是一个业务代码里有个低概率触发的死循环，
> 在特定数据条件下会一直循环不退出，通过 jstack 看到这个线程一直是 RUNNABLE 状态
> 且调用栈没有变化，确认是死循环，修复了退出条件后解决。

**面试官：内存泄漏怎么排查？**

> 内存泄漏的特征是 Full GC 越来越频繁但老年代使用率 GC 后回落越来越少，
> 最终 OOM。排查时先用 `jstat -gcutil <pid>` 观察老年代趋势，
> 确认是泄漏后用 `jmap -dump` 导出堆快照，
> 再用 MAT 工具分析 Dominator Tree，找到持有最多内存的对象，
> 顺着引用链找到是哪里的代码一直持有引用没有释放。
